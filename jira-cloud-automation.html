Overview

What I automate in Jira Cloud

I build automation around Jira Cloud to support teams during large-scale migrations, cleanups, and day-to-day operations. The goal is to transform fragile, manual admin work into repeatable, traceable, and safe jobs that can run on schedule or on demand.

The examples in this portfolio are simplified and anonymized, but they reflect real scenarios: custom field inventories, cleanup orchestration, and sample issue generation for demos and testing.

Instance discovery & inventory Custom field lifecycle Scheduled checks & alerts Guardrails for migrations

Automation Scope

Types of jobs I design

1. Discovery & Inventory

ScriptRunner jobs and Python scripts that scan Jira Cloud configuration and export structured inventories (custom fields, screens, workflows, projects).

  • Export metadata to CSV / JSON for analysis
  • Tag risky or unused configuration elements
  • Support migration planning and cleanups

2. Governance & Guardrails

Automation that keeps configuration in a good state: enforcing naming conventions, validating new fields, and detecting violations over time.

  • Rules for “read-only” projects during migrations
  • Checks to prevent duplicate or conflicting fields
  • Lightweight alerts instead of manual audits

3. Scheduled Jobs & Demo Data

Jobs that generate issues on a schedule for demo or monitoring purposes.

  • Weekly Issue Generator for demo projects
  • Structured summaries stored in custom fields or comments
  • Logging and idempotency to avoid duplicates

4. Integration Helpers

Small services around Jira APIs (Python) that support bulk operations when ScriptRunner alone is not enough.

  • Batch updates to issues or fields via REST
  • Export/import helpers for configuration snapshots
  • Alignment with GitHub for version control

Automation Architecture

How the pieces work together

I combine Jira Cloud automation, ScriptRunner (Cloud), and external Python scripts into a simple but powerful architecture. The exact design depends on the company’s constraints, but the pattern follows the same logic:

  • 1. Configuration discovery ScriptRunner · REST API · CSV / JSON exports
    Jobs that read Jira configuration (fields, screens, projects) and store a snapshot for analysis and traceability.
  • 2. Data analysis & rules Python · Notebooks · Ad-hoc validation
    External scripts that analyze inventories, detect patterns (duplicates, unused, risky), and generate rule sets or candidate changes.
  • 3. Safe changes & guardrails ScriptRunner jobs · Dry-run modes
    Carefully designed jobs that apply changes in a controlled way: dry-run first, verbose logging, rollback notes, and clear scope.
  • 4. Monitoring & demos Scheduled issues · Alerts · Dashboards
    Scheduled “heartbeat” issues and checks that prove the automations are working and keep admins informed about the instance health.
Design principle: every job should be observable. If an admin cannot tell what changed, when it ran, and what it touched, it’s not ready to run against an enterprise Jira instance.

Highlighted Automations

Selected examples from this track

Custom Fields Inventory

ScriptRunner job that exports a detailed inventory of custom fields, including usage, contexts, and comments for fields that couldn’t be fully resolved.

  • Supports custom field cleanups and consolidation
  • Feeds downstream scripts for rules & decisions

Weekly Issue Generator

Scheduled job that creates demo issues in a dedicated project to showcase automation behavior and keep the environment “alive” for stakeholders.

  • Demonstrates scheduling, logging, and safe defaults
  • Useful for guided demos or training environments
  • Documented in: weekly-issue-generation.html

Read-Only Projects (concept)

Conceptual design for rules that temporarily enforce read-only behavior in specific projects during migrations or critical windows.

  • Prevents accidental changes during cut-overs
  • Can be implemented via ScriptRunner + issue transitions
  • Aligns with governance & change-management policies

Field Cleanup Orchestrator (concept)

Orchestrated approach where inventory, analysis, and change jobs communicate via structured outputs (CSV/JSON) instead of one big “god script”.

  • Smaller, testable jobs instead of a single risky script
  • Supports phased cleanups and controlled rollouts
  • Ready to be extended per client or instance

Delivery & Impact

How this translates to value

The main impact of this automation track is risk reduction and time saved for Jira admins and project teams. Instead of relying on one-off scripts or manual clicks, admins get tools they can run repeatedly and trust.

  • Faster assessments of Jira configuration before migrations
  • Fewer surprises related to “hidden” or unused custom fields
  • Safer change windows thanks to read-only and validation patterns
  • Better communication with stakeholders through documented jobs
Next step: connect these automations with CI (GitHub Actions) so ScriptRunner code, Python helpers, and documentation live together under version control and can be promoted between environments.
