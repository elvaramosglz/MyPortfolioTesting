Jira Cloud Automation

I design and implement automation for Jira Cloud using Groovy (ScriptRunner) and Python (REST API), focusing on reducing manual work, improving data integrity, and supporting large-scale administration and migrations.

Jira Cloud ScriptRunner REST API Automation Admin Tools

Groovy Scripts · ScriptRunner for Jira Cloud

These scripts run directly in Jira Cloud using ScriptRunner. They are ideal for workflow automation, validations, scheduled jobs, and portfolio-friendly demos that live entirely inside the Atlassian ecosystem.

  1. 1. Weekly Issue Generator →

    ScriptRunner scheduled job that automatically creates a demo issue in a Jira Cloud project. It demonstrates scheduled automation, REST API calls, and logging best practices.

    • Creates a new issue according to a configured schedule (daily, weekly, etc.).
    • Uses Groovy + ScriptRunner + /rest/api/3/issue.
    • Builds an ADF description with a link back to this portfolio page.

Python Scripts · Jira Cloud REST API

These examples use Python to call the Jira Cloud REST API for administration, reporting, and bulk operations. They complement the Groovy/ScriptRunner work by focusing on configuration inventory and data extraction outside Jira.

  1. 1. Jira Cloud Custom Fields Inventory →

    Python script that connects to Jira Cloud and builds a full inventory of custom fields, their contexts, and their option values, exporting everything to JSON and CSV.

    • Uses /rest/api/3/field, field context, and option endpoints.
    • Resolves option lists via context APIs and editmeta fallback.
    • Runs via GitHub Actions with Jira credentials stored as encrypted secrets.
  2. (Planned) Additional Jira Cloud admin tooling

    Additional Python utilities are planned to extend this toolkit.

    • Create and update custom fields via REST.
    • Bulk update issues based on CSV inputs and business rules.
    • Export workflows, permission schemes, and configuration for documentation.

Source code: The underlying Groovy and Python scripts are kept in dedicated repositories and are designed so that sensitive data (Jira URLs, emails, API tokens) is always injected via environment variables and GitHub Actions secrets.
